# 全栈 A 后续工作计划

## 1. 文档目的

本文面向全栈 A，用于把后续开发工作按功能模块、优先级、协作边界和验收口径拆开，方便后续逐项推进。

全栈 A 的主责是触达增长链路：

```text
模板 -> 规则 -> 事件 -> 任务 -> 发送 -> 回执 -> 短链 -> 统计
```

一句话边界：

> 负责短信怎么被配置、触发、发送、追踪和分析。

## 4. 分模块工作计划

### 4.1 运营总览

目标：把首页从普通后台 Dashboard 调整为增长运营总览。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 校验核心 KPI 口径 | 发送量、成功量、失败量、拦截量、待执行任务、点击量、CTR 口径一致 |
| P0 | 展示运营状态 | 启用规则、待执行任务、异常任务、拦截量 |
| P1 | 场景表现模块 | 注册转化、会员召回、活动通知、售后回访卡片 |
| P1 | 最近发送改版 | Activity Feed 或精简发送记录 |
| P1 | UI/UX 重构 | Hero KPI、轻量卡片、增长运营产品气质 |

验收标准：

1. 打开首页即可看出当前触达链路健康度。
2. KPI 与发送记录、任务数据能对上。
3. 页面第一眼是营销自动化平台，不是传统管理后台。

### 4.2 模板中心

目标：让运营可以可靠创建、管理、预览和测试短信模板。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 模板列表可读性优化 | 展示名称、场景、Provider 模板 Code、变量、状态 |
| P0 | 新建模板表单完善 | 必填校验、变量格式、状态选择 |
| P1 | 模板详情和编辑 | Modal 或独立详情页 |
| P1 | 模板变量预览 | 输入变量后预览短信内容 |
| P1 | 模板测试发送 | 选择模板和手机号，调用测试发送或手动发送链路 |
| P1 | 模板卡片化 UI | 场景标签、状态 Pill、轻量操作区 |

验收标准：

1. 模板可创建、启停、查看详情。
2. 停用模板不可被新规则误用。
3. 测试发送必须走安全校验和操作日志契约。

### 4.3 规则中心

目标：把规则中心做成 A 的核心生产力页面，让运营能看懂、能测试、能安全启停。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 结构化规则表单 | 事件、延迟、条件、模板按结构化控件配置 |
| P0 | 条件类型组件化 | `none`、`not_purchased_membership` 先闭合，预留其他条件 |
| P0 | 规则启停确认 | 启用、停用前二次确认 |
| P0 | 规则详情 | 展示触发事件、条件、模板、任务和发送表现 |
| P1 | 规则测试 | 输入模拟事件，预览匹配规则和生成任务 |
| P1 | 条件自然语言预览 | 例如“注册后 24 小时仍未购买会员则发送” |
| P1 | 规则复制 | 复制已有规则快速创建 |
| P1 | 影响范围预估 | 展示命中事件数、预计任务数或风险提示 |
| P1 | 高风险审批入口 | A 发起业务动作，B 管审批流程 |

验收标准：

1. 运营不需要手填条件字符串。
2. 任一规则都能解释“什么时候触发、满足什么条件、发送哪个模板”。
3. 启停规则不会绕过权限、审批和操作日志要求。

### 4.4 手动发送

目标：支持测试、补发和小范围手动触达，但不能绕过治理安全。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 发送表单完善 | 模板选择、手机号输入、变量输入、发送预览 |
| P0 | 发送前安全校验展示 | 白名单、黑名单、退订、频控、安静时段、Provider 状态 |
| P0 | 发送结果展示 | 成功、失败、拦截原因、发送记录入口 |
| P1 | 多手机号输入预留 | 暂不做完整人群包，但结构预留批量入口 |
| P1 | 操作确认 | 真实 Provider 或非测试号码场景二次确认 |

验收标准：

1. 手动发送不能绕过 B 提供的 `POST /api/safety/send-check` 或等价 helper。
2. 发送失败或拦截必须能看到原因。
3. 发送后能追踪到发送记录、回执和短链。

### 4.5 事件触发和事件流水

目标：让运营和研发能从事件入口追踪到规则、任务和发送结果。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 事件列表完善 | eventId、事件类型、手机号、来源、状态、失败原因 |
| P0 | 模拟事件能力回归 | 四类事件均可模拟触发 |
| P0 | 事件详情 | 原始 payload、匹配规则、生成任务、处理结果 |
| P1 | payload 摘要展示 | 列表展示关键字段，不直接铺满 JSON |
| P1 | 事件到任务追踪 | 从 eventId 跳到关联任务和发送记录 |

验收标准：

1. 四类事件均可接收或模拟：`user_register`、`membership_expired`、`campaign_start`、`order_completed`。
2. 重复 eventId 不会重复生成任务。
3. 任一事件都能解释是否匹配规则、是否生成任务。

### 4.6 任务中心

目标：任务中心要回答三个问题：什么时候发、为什么发、为什么没发。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 任务状态展示统一 | `pending`、`sending`、`success`、`failed`、`blocked`、`skipped`、`cancelled` |
| P0 | 任务详情 | 事件、规则、模板、条件结果、发送记录 |
| P0 | 跳过原因展示 | 已购买、黑名单、频控、规则停用、条件错误等 |
| P0 | 失败重试 | 单任务失败重试 |
| P0 | 待发送取消 | 单任务取消 |
| P1 | 状态分组视图 | 待执行、异常、已完成分组 |
| P1 | 批量取消和批量重试 | A 定义任务语义，B 记录批次、权限和审计 |
| P1 | 执行到期任务体验优化 | 展示本次处理数量和失败原因 |

验收标准：

1. 任务状态流转清晰，新增状态前必须同步 B。
2. 条件判断结果、跳过原因和失败原因可见。
3. 取消和重试不会破坏发送幂等。

### 4.7 发送记录

目标：发送记录承担排查、回执、短链和复盘入口。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 列表可读性优化 | 手机号脱敏、触发方式、模板、规则、状态、Provider |
| P0 | 发送详情 | 请求参数、Provider 返回、回执、短链、点击 |
| P0 | 回执状态展示 | submitted、delivered、failed 等映射清晰 |
| P1 | 筛选能力完善 | 时间、场景、规则、模板、状态、Provider |
| P1 | 模拟回执或标记送达 | 测试环境辅助联调 |
| P1 | 点击日志关联 | 从发送记录查看短链点击 |

验收标准：

1. 从一条发送记录能定位到任务、规则、模板、事件。
2. Provider 返回和回执状态可排查。
3. 手机号展示符合脱敏规则。

### 4.8 回执与短链

目标：保证发送后的状态更新、点击追踪和统计数据可靠。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 回执写入稳定 | Provider callback 写入 `sms_receipt` 并更新发送状态 |
| P0 | 回执幂等 | 避免重复 callback 重复写入 |
| P0 | 短链跳转稳定 | `GET /s/{shortCode}` 记录点击并跳转 |
| P1 | 点击数据关联 | 点击日志关联 task、user、rule、template |
| P1 | 正式 Provider 回执映射预留 | 为正式营销短信字段差异留扩展点 |

验收标准：

1. 回执重复上报不会导致重复统计。
2. 短链点击能进入 CTR 统计。
3. 回执、短链、发送记录之间可互相追踪。

### 4.9 统计分析

目标：从“能发短信”升级到“能复盘触达效果”。

| 优先级 | 工作项 | 交付物 |
| --- | --- | --- |
| P0 | 概览统计回归 | 发送、成功、失败、拦截、点击、CTR |
| P1 | 按规则统计 | 每条规则任务数、发送数、成功数、点击数、CTR |
| P1 | 按模板统计 | 每个模板发送表现 |
| P1 | 按场景统计 | 注册转化、会员召回、活动通知、售后回访 |
| P1 | 趋势图 | 日发送量、成功率、点击率 |
| P1 | 漏斗图 | 任务 -> 发送 -> 送达 -> 点击 |
| P2 | 转化归因预留 | 预留 `membership_purchased` 等事件回流口径 |

验收标准：

1. 统计口径与任务、发送记录、点击日志一致。
2. 至少能按规则、模板、场景定位效果差异。
3. 图表保持轻量，不做复杂 BI。

## 6. 与全栈 B 的协作点

以下事项必须提前同步，不建议单独改：

| 协作点 | A 负责 | B 负责 |
| --- | --- | --- |
| 权限体系 | 页面和按钮接入权限点 | 权限点、角色、接口鉴权 |
| 发送前安全校验 | 所有发送入口调用并展示结果 | 白名单、黑名单、退订、频控、安静时段 |
| 操作日志 | 定义业务动作和对象 | 日志写入、查询和详情 |
| 批量任务 | 定义取消、重试语义 | 批次记录、明细、权限、审计 |
| 审批流程 | 发起高风险规则启停或真实发送动作 | 审批单、审批处理、审批状态 |
| 事件来源 | 事件入库、规则匹配、任务生成 | appId、secret、HMAC、接入日志 |
| 状态枚举 | 任务、发送、规则业务状态 | 全局状态展示和权限联动 |
| 菜单和权限 key | A 页面菜单项和按钮 | AppShell、菜单框架、权限 key |

## 7. 不应主改的区域

全栈 A 不应主改以下模块，除非是联调中双方确认的契约接入：

1. 登录、注册申请、忘记密码、设置密码。
2. 用户管理、角色权限、权限点定义。
3. 白名单、黑名单、退订管理页面和接口。
4. 发送控制配置页面和接口。
5. 事件来源管理和接入日志。
6. 操作日志、导出任务、批量操作列表、审批记录页面。
7. `apps/api/src/modules/governance/governance.service.js` 中账号、权限、安全治理、审计流程的主逻辑。

## 8. 推荐执行顺序

### 阶段一：主链路稳定和现状回归

1. 回归模板创建、启停。
2. 回归规则创建、启停。
3. 回归四类事件模拟和事件幂等。
4. 回归任务生成、到期执行和 worker。
5. 回归手动发送、发送记录、回执、短链。
6. 对齐任务状态、发送状态、统计口径。

阶段目标：确认现有测试版闭环没有被后续改造破坏。

### 阶段二：关键详情和状态解释

1. 补模板详情。
2. 补规则详情。
3. 补事件详情。
4. 补任务详情。
5. 补发送记录详情。
6. 补跳过原因、失败原因、Provider 返回、回执信息。

阶段目标：从 eventId 或 taskId 能追踪完整链路。

### 阶段三：规则和任务生产化

1. 规则结构化条件表单。
2. 规则自然语言预览。
3. 规则测试。
4. 规则启停二次确认。
5. 任务取消。
6. 任务重试。
7. 批量取消、批量重试与 B 联调。

阶段目标：运营能安全配置和维护自动触达规则。

### 阶段四：A 页面 UI/UX 重构

1. 总览 Hero KPI 和场景表现。
2. 模板中心卡片化。
3. 规则中心自动化规则卡片。
4. 手动发送预览体验。
5. 任务中心状态分组。
6. 发送记录可读性优化。

阶段目标：A 主责页面符合 Marketing Automation Platform 风格。

### 阶段五：统计分析升级

1. 按规则统计。
2. 按模板统计。
3. 按场景统计。
4. 趋势图。
5. 漏斗图。
6. 转化归因字段预留。

阶段目标：支持基础运营复盘。

### 阶段六：上线前联调

1. 和 B 联调登录态、菜单权限、按钮权限。
2. 和 B 联调发送前安全校验。
3. 和 B 联调关键写操作日志。
4. 和 B 联调高风险审批。
5. 和 B 联调批量任务。
6. 验证真实 Provider 只能在白名单和安全开关下触发。

阶段目标：小范围灰度前，触达链路、安全链路和审计链路闭合。

| M6 | 上线前联调 | 权限、安全校验、操作日志、审批、批量任务联调完成 |

## 10. 每次开发前检查清单

1. 是否修改了 A/B 共享文件，例如 `menus.tsx`、`App.tsx`、`api.ts`、`types/index.ts`。
2. 是否新增或修改了任务状态、发送状态、规则状态。
3. 是否影响手机号存储、脱敏或查询规则。
4. 是否影响发送前安全校验。
5. 是否需要操作日志。
6. 是否需要权限点或按钮权限。
7. 是否需要审批。
8. 是否会影响批量任务。
9. 是否修改了 Prisma 表结构。
10. 是否需要同步接口文档和测试用例。
